459/// Returns the execution input to the executed contract and decodes it as `T`.
460///
461/// # Note
462///
463/// - The input is the 4-bytes selector followed by the arguments of the called function
464///   in their SCALE encoded representation.
465/// - No prior interaction with the environment must take place before calling this
466///   procedure.
467///
468/// # Usage
469///
470/// Normally contracts define their own `enum` dispatch types respective
471/// to their exported constructors and messages that implement `scale::Decode`
472/// according to the constructors or messages selectors and their arguments.
473/// These `enum` dispatch types are then given to this procedure as the `T`.
474///
475/// When using ink! users do not have to construct those enum dispatch types
476/// themselves as they are normally generated by the ink! code generation
477/// automatically.
478///
479/// # Errors
480///
481/// If the given `T` cannot be properly decoded from the expected input.
482pub fn decode_input<T>() -> core::result::Result<T, DispatchError>
483where
484    T: DecodeDispatch,
485{
486    <EnvInstance as OnInstance>::on_instance(|instance| {
487        EnvBackend::decode_input::<T>(instance)
488    })
489}

721/// Prefer this over [`is_contract`] when checking whether your contract is being called
722/// by a contract or a plain account. The reason is that it performs better since it does
723/// not need to do any storage lookups.
724///
725/// A return value of `true` indicates that this contract is being called by a plain
726/// account. and `false` indicates that the caller is another contract.
727///
728/// # Errors
729///
730/// If the returned value cannot be properly decoded.
731pub fn caller_is_origin<E>() -> bool
732where
733    E: Environment,
734{
735    <EnvInstance as OnInstance>::on_instance(|instance| {
736        TypedEnvBackend::caller_is_origin::<E>(instance)
737    })
738}

760/// Replace the contract code at the specified address with new code.
761///
762/// # Note
763///
764/// There are a few important considerations which must be taken into account when
765/// using this API:
766///
767/// 1. The storage at the code hash will remain untouched.
768///
769/// Contract developers **must ensure** that the storage layout of the new code is
770/// compatible with that of the old code.
771///
772/// 2. The contract address (`AccountId`) remains the same, while the `code_hash` changes.
773///
774/// Contract addresses are initially derived from `hash(deploying_address ++ code_hash ++
775/// salt)`. This makes it possible to determine a contracts address (`AccountId`) using
776/// the `code_hash` of the *initial* code used to instantiate the contract.
777///
778/// However, because `set_code_hash` can modify the underlying `code_hash` of a contract,
779/// it should not be relied upon that a contracts address can always be derived from its
780/// stored `code_hash`.
781///
782/// 3. Re-entrant calls use new `code_hash`.
783///
784/// If a contract calls into itself after changing its code the new call would use the new
785/// code. However, if the original caller panics after returning from the sub call it
786/// would revert the changes made by `set_code_hash` and the next caller would use the old
787/// code.
788///
789/// # Errors
790///
791/// todo: this enum variant no longer exists
792/// `ReturnCode::CodeNotFound` in case the supplied `code_hash` cannot be found on-chain.
793///
794/// # Storage Compatibility
795///
796/// When the smart contract code is modified,
797/// it is important to observe an additional virtual restriction
798/// that is imposed on this procedure:
799/// you should not change the order in which the contract state variables
800/// are declared, nor their type.
801///
802/// Violating the restriction will not prevent a successful compilation,
803/// but will result in the mix-up of values or failure to read the storage correctly.
804/// This can result in severe errors in the application utilizing the contract.
805///
806/// If the storage of your contract looks like this:
807///
808/// ```ignore
809/// #[ink(storage)]
810/// pub struct YourContract {
811///     x: u32,
812///     y: bool,
813/// }
814/// ```
815///
816/// The procedures listed below will make it invalid:
817///
818/// Changing the order of variables:
819///
820/// ```ignore
821/// #[ink(storage)]
822/// pub struct YourContract {
823///     y: bool,
824///     x: u32,
825/// }
826/// ```
827///
828/// Removing existing variable:
829///
830/// ```ignore
831/// #[ink(storage)]
832/// pub struct YourContract {
833///     x: u32,
834/// }
835/// ```
836///
837/// Changing type of a variable:
838///
839/// ```ignore
840/// #[ink(storage)]
841/// pub struct YourContract {
842///     x: u64,
843///     y: bool,
844/// }
845/// ```
846///
847/// Introducing a new variable before any of the existing ones:
848///
849/// ```ignore
850/// #[ink(storage)]
851/// pub struct YourContract {
852///     z: Vec<u32>,
853///     x: u32,
854///     y: bool,
855/// }
856/// ```
857///
858/// Please refer to the
859/// [Open Zeppelin docs](https://docs.openzeppelin.com/upgrades-plugins/1.x/writing-upgradeable#modifying-your-contracts)
860/// for more details and examples.
861#[cfg(feature = "unstable-hostfn")]
862pub fn set_code_hash<E>(code_hash: &H256) -> Result<()>
863where
864    E: Environment,
865{
866    <EnvInstance as OnInstance>::on_instance(|instance| instance.set_code_hash(code_hash))
867}
